# 核心工作台前端重组设计

日期：2026-07-16

## 背景

当前工作台已经覆盖作品管理、规划、故事圣经、大纲、章节、时间线、质检、AI 生成、发布和备份，但主要前端逻辑集中在 `src/App.tsx`，用户需要在多个页面之间切换才能完成一次章节创作。此次改造采用“大规模前端重组”路线：保留现有 Fastify API、SQLite 数据和版本机制，重做进入作品后的信息架构与章节创作工作区。

## 目标

- 将章节创作设为进入作品后的默认主任务。
- 在同一工作区完成章节目标确认、上下文查看、AI 生成、质检、修改和版本接受。
- 让人物、事实、伏笔、时间线和前后章摘要按当前章节相关性呈现。
- 统一加载、错误、保存、生成中、候选稿和质检状态。
- 桌面端提供稳定的三栏工作区，移动端提供抽屉式上下文和底部导航。
- 在不迁移数据库、不破坏现有 API 的情况下逐步替换旧页面。

## 非目标

- 本阶段不重做数据库 schema。
- 本阶段不更换 AI 供应商、提示词协议或生成服务。
- 本阶段不增加新的出版平台适配。
- 本阶段不一次性删除旧页面；旧页面作为兼容入口保留，待新工作区稳定后再迁移。

## 信息架构

作品入口进入 `WorkbenchShell`，默认打开当前待创作章节。页面结构分为：

1. 顶部栏：作品切换、卷/弧/章节位置、保存状态、全局搜索和设置入口。
2. 左侧导航：卷、弧、章节列表、章节状态、伏笔提醒。支持折叠。
3. 中央工作区：章节标题、目标摘要、正文编辑器、候选稿/历史版本/差异视图、质检结果。
4. 右侧上下文抽屉：人物、事实、相关伏笔、时间线、前后章摘要和上下文来源。默认只展开摘要，详情按需打开。
5. 移动端：顶部作品/章节选择器，中央编辑区，底部导航切换“写作、上下文、质检、更多”。

规划、圣经、时间线和发布仍保留独立页面，但从当前章节工作区提供上下文入口；这些页面不再承担章节创作主导航。

## 前端模块边界

建议按功能拆分 `src/`：

```text
src/
  app/              路由、查询客户端、全局布局
  components/       Modal、Toast、Button、状态和空状态
  features/
    workbench/      WorkbenchShell、导航和响应式布局
    chapters/       编辑器、候选稿、版本、差异、自动保存
    context/        人物、事实、伏笔、时间线和章节摘要
    quality/        预检、AI 质检、问题定位和修复建议
    outline/        规划、大纲和弧线辅助面板
    publishing/     发布记录和导出
    settings/       模型、提示词、快照和调用记录
  hooks/            工作区、章节、自动保存和生成任务 hooks
  lib/              API 客户端、格式化、错误和键盘快捷键
```

现有 API 调用通过 `src/api.ts` 统一封装。新组件只依赖功能 hooks 或明确的共享类型，不直接读取 React Router 参数或拼接 API 路径。

## 核心数据流

打开作品时加载现有 workspace 数据，并单独查询当前章节的工作草稿、版本和质检状态。章节切换时：

1. 保存当前工作草稿到恢复区。
2. 加载目标章节及其相关上下文。
3. 以章节 ID 作为所有生成、质检和候选稿查询的作用域。

点击生成时先显示上下文摘要和预计使用的来源，再创建生成任务。流式结果只写入候选稿，不覆盖当前正式正文。用户接受候选稿后才创建正式版本并刷新章节状态。质检分为硬冲突、提醒和风格建议；硬冲突阻止接受，其他问题允许用户明确忽略。

## 兼容与迁移

- 保留 `/novel/:id/:view?` 路由形式，新工作台接管默认入口。
- 旧的 `planning`、`canon`、`outline`、`timeline`、`publish`、`settings` 页面先通过导航继续可达。
- API 返回结构保持兼容；必要的聚合只在前端 hooks 中完成。
- 分阶段迁移：先壳层和章节页，再上下文抽屉，再 AI/质检面板，最后迁移辅助页面。
- 任一阶段失败时可回退到旧页面入口，不改变已有数据。

## 错误与状态

- 所有请求统一显示加载、空数据、重试和错误详情。
- 自动保存失败时保留本地编辑内容，并显示明确的恢复操作。
- 生成中允许停止；停止或失败时保留已产生的部分候选稿并标记状态。
- 切换章节或关闭页面前，如有未保存变更，显示非破坏性确认。
- 删除、恢复快照和接受正式版本继续使用应用内确认弹窗。

## 验收标准

- 新用户进入作品后无需访问其他页面即可完成：查看章节目标、查看相关上下文、生成候选稿、运行质检、接受版本。
- 桌面端三栏在常见宽度下不发生内容重叠；移动端可通过底部导航访问同样四个核心区域。
- 候选稿生成永不覆盖当前正式正文；接受操作可在版本历史和差异视图中追溯。
- 上下文面板展示来源和更新时间，且能定位到相关人物、事实或伏笔。
- 网络错误、生成失败、自动保存失败和空数据都有可操作反馈。
- 现有 Playwright 核心流程、服务器单测和生产构建通过。

## 分阶段交付

1. 提取共享 UI、统一状态和错误处理，建立新的 `WorkbenchShell`。
2. 迁移章节编辑、候选稿、版本和差异视图。
3. 实现上下文抽屉和章节相关性展示。
4. 将 AI 生成、停止、质检和问题定位集成到工作区。
5. 迁移规划、圣经、时间线、发布和设置为辅助面板，完成响应式和回归测试。
