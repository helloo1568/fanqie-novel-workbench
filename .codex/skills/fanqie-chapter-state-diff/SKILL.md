# 章节功能与状态差异测试

## R — 原文（Reading）

> 读者看完一章只用一句话概括，却没有实际性的故事剧情推进，这就是太慢了。
>
> — 《单章到底怎么改？》第180—192行

## I — 方法论骨架（Interpretation）

章节价值可以用状态差异而不是文笔印象判断。先写章前状态，再写章后在目标、关系、资源、认知、风险、时间、地点或情绪上的变化。至少一项变化必须对后续选择有影响。

“过渡”不是豁免：移动、休息和日常场景可以改变关系、提供信息、制造下一期待或结算上一事件。若删掉整章不影响任何后续条件，它应被合并、压缩或重写。

## A1 — 资料集中的应用（Past Application）

### 案例 1：逐章数据定位后改文
- **问题**：长篇章节数据表现不稳，却不知道问题在哪
- **方法论的使用**：编辑逐章分析，把异常章节对应到具体文本，再要求定向改文
- **结论**：章节成为可单独检验的功能单元
- **结果**：作者后续长篇收入改善，但编辑资源是混杂变量

### 案例 2：大纲骨架上的单章修改
- **问题**：章节表达、节奏与全书结构容易混改
- **方法论的使用**：课程把章节视为骨架上的血肉，先判断能否局部修复
- **结论**：状态和核心信息不变时可局部改章
- **结果**：用于建立候选版本而非覆盖原稿

## A2 — 触发场景（Future Trigger）

### 用户会在什么情境下需要这个 skill？

1. 一章写完不知道发生了什么
2. 过渡章删掉似乎也不影响后文
3. 需要审核章纲是否足够具体

### 语言信号

- “这章有必要吗”
- “有没有推进”
- “章前章后差别”
- “chapter function / state diff”

### 与相邻 skill 的区分

- 与 fanqie-pace-triad 的区别：本 skill 先判断有没有有效变化；后者判断推进、信息、情绪比例。
- 与 fanqie-revision-evidence-loop 的区别：本 skill 诊断章节功能；后者选择改动层级和验证版本。

## E — 可执行步骤（Execution）

1. **写章节任务**
   - 动作：用一句话写本章必须改变什么，以及这项改变怎样服务当前弧。
   - 完成标准：任务包含对象、变化和后续影响。

2. **生成状态差异**
   - 动作：列章前/章后的目标、关系、资源、认知、风险、时间和情绪。
   - 完成标准：至少一项有可引用正文证据的有效差异。

3. **检查场景贡献**
   - 动作：给每个场景标记推进、人物、信息、情绪、伏笔或结算功能。
   - 完成标准：每场至少一项主功能；纯重复场景被标出。

4. **决定处置**
   - 动作：无差异则删除/合并；差异弱则补选择与后果；差异成立则保留。
   - 完成标准：处置理由明确，且不会破坏已确认事实。

## B — 边界（Boundary）

### 不要在以下情况使用

- 诗性氛围或情感恢复章可以用情绪/关系变化通过，不要求物质奖励。
- 不要把状态项数量当质量分数；一个关键变化可能足够。
- 只做事实诊断时不自动重写正文。

### 资料集警告的失败模式

- 把过渡章写成无信息位移。
- 集中写景让剧情暂停。
- 同义复述与流程对白制造机械感。

### 证据局限

- 官方课程覆盖 2021—2026 年，市场、产品和平台规则必须按日期复核。
- 成功作者案例存在幸存者偏差，方法用于诊断和降低风险，不承诺流量或收益。
- 视频课程部分来自本地 ASR 转写，重要术语与硬规则需回查原始页面或多源证据。
